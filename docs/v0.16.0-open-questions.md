## 1. Where does the subnet router run?

Three options, each with different operational cost and user
friction. The headscale 0.23+ `tagOwners` mechanism allows any of
them â€” the choice is operational, not technical.

| Option | What | Pros | Cons |
|--------|------|------|------|
| **(a) User's own machine** | Each user runs `tailscale up --advertise-routes=10.0.<uid>.0/24` on a machine they already own (laptop, NAS, home server, VPS). | Zero skygate-side cost. The user can choose where their subnet lives geographically. | Requires the user to keep a machine on 24/7. Some users may not have one. |
| **(b) Skygate-managed Docker sidecar per user** | Skygate spins up a tiny `tailscale/tailscale` container per user on the skygate host, advertises their CIDR, persists state. | Always-on, no user effort, no remote machine needed. | Per-user sidecar = NĂ—resource overhead on the skygate host. State needs to persist across skygate restarts. |
| **(c) Shared skygate-side router, one router per user via `tag:subnet-router` for the tag and a Tailscale ACL that allows the user to advertise their personal CIDR through it** | Skygate runs one router container; each user gets a per-user namespace and the router isolates them via the ACL `tagOwners` rule. | Single resource overhead; ACL enforces isolation. | Router is a SPOF for the user; users cannot put services "at home". |

The user's current request: keep the `tag:exit-node` mesh global
and add per-user subnets. **Recommendation: start with (a)**
because it's free and tests the schema. Fall back to (b) for
customers who don't have a 24/7 machine. (c) is the SaaS-grade
option for "1000+ users, no per-user ops".

## 2. CIDR strategy

| Choice | Capacity | Notes |
|--------|----------|-------|
| `/24` per user in `10.0.<uid>.0/24` | 256 users | Plenty for the current operator (~10 users). Each user has 254 usable IPs in their subnet. |
| `/28` per user in `10.0.<uid>.<sub>.0/28` | 4096 users (256 subnets per /24) | More granularity. Each user has 14 usable IPs in their subnet. Enough for "user's main machine + 12 devices + 1 router". |
| `/56` per user (IPv6) or `/48` per user | essentially infinite | If we want to go IPv6-heavy. But headscale subnet-router advertising is mostly IPv4 today. |

**Recommendation: `/24` per user** for v0.16.0, with the schema
flexible enough to support a `subnet_bits` column so we can
move to `/28` without a schema migration. Operator's call.

## 4. Multi-plane alignment (v0.12.0 compatibility)

v0.12.0 already supports per-user headscale control plane
(`portal_users.headscale_url` + `headscale_api_key_enc`).
v0.16.0 must respect this:

- The `user_subnets` table needs a `control_plane_url` column
  (FK to the plane's URL) so each user's subnet lives in
  THEIR plane's CIDR space
- Different planes use different CIDR spaces (no cross-plane
  subnet sharing â€” that wouldn't work without VPN-mesh)
- `GenerateACL()` becomes "per plane" already in v0.13.0
  (separate from per-subnet), so v0.16.0's per-subnet rules
  need to be generated in the per-plane context

**No question here, just a constraint:** the per-user bot routing
(v0.12.1) and per-plane ACL (v0.13.0) MUST be done before
v0.16.0 because the per-subnet feature inherits the per-plane
context.

## 5. Migration path

Two practical options for the ~10 existing users on the
operator's tailnet:

| Option | What | Pros | Cons |
|--------|------|------|------|
| **(A) Opt-in, per user** | Each existing user keeps their flat `100.64.0.0/10` IP until they explicitly opt in via `/admin/users/{id}/subnet`. When they opt in, skygate creates a subnet router AND keeps the flat device. | Zero migration risk. User can phase out the flat device at their own pace. | Two devices per user temporarily (the flat one + the subnet router). |
| **(B) Forced re-create** | Delete the user's flat device, recreate as subnet router + new tag. | Clean state. | Breaks every running client. User must re-auth. |

**Recommendation: (A).** Same principle as v0.12.0 â€” the
operator explicitly chose "no forced migration, opt-in
per user". Don't break that for v0.16.0.

## 6. Cross-user subnet sharing (v0.17.1)

When user A shares `10.0.42.0/24` with user B, what's the
DNS / ACL shape?

- ACL: `{src: ["<user_B>@tsnet"], dst: ["<user_A_subnet>:*"]}`
  (headscale identity names with the @-form)
- DNS: `skygate-<user_B>.tailnet.skynas.ru` resolves to user_B's
  subnet router, not user_A's. To reach user_A's shared services,
  user_B uses `10.0.42.x:port` directly (via Tailscale routing),
  OR skygate creates a headscale "service" record that maps
  `<user_A_service>.tailnet.skynas.ru` â†’ `10.0.42.x:port`
  (headscale 0.23+ feature; v0.19.0)
- Audit log: every share/revoke is a `subnet_share_granted` /
  `subnet_share_revoked` row

**Recommendation: ship v0.17.1 with IP-level sharing only**
(CIDR ACL rule). Defer the service-record DNS to v0.19.0.

## 7. Headscale API surface for subnet routers

Headscale 0.23+ supports:
- `POST /api/v1/node/{id}/routes` â€” set advertised routes
- `tagOwners` in the ACL â€” `tag:subnet-router: ["skyadmin@tsnet"]`
  (only the operator can tag a node with `tag:subnet-router`)

For v0.16.1, the workflow is:
1. Skymate creates a preauth key tagged with `tag:subnet-router`
   for the user
2. User runs `tailscale up --login-server=... --authkey=...
   --advertise-routes=10.0.<uid>.0/24 --accept-routes`
   on their machine
3. Skymate sees the new node in headscale via the
   `AutoSync` mechanism, tags it `tag:subnet-router` (the
   user can't, because the tag is in `tagOwners` for
   skyadmin only)
4. Skymate calls `ApproveRoutes(node, [10.0.<uid>.0/24, 0.0.0.0/0, ::/0])`
5. The user's clients, with `--accept-routes`, now see
   the personal subnet AND can route through any
   `tag:exit-node`

**No question, just a note:** the `tag:subnet-router` tag is
new and must be added to `tagOwners` in `GenerateACL()` in
v0.17.0.

## 8. Testing strategy

Three layers needed:
1. **Unit tests** for the CIDR allocator (no overlaps, no
   out-of-range, idempotent)
2. **Integration test** with a fake headscale (v0.16.1+):
   create user, create preauth key, simulate node registration
   + route approval, verify ACL
3. **Live smoke** on the operator's tailnet: opt-in the
   existing 10 users one by one, verify each has both
   `100.64.0.x` AND `10.0.<uid>.0/24` reachable

**Recommendation: start with unit tests + a fake-headscale
integration test in v0.16.0 (before touching the live tailnet)**.
Live opt-in is per-user behind a "Try it" button on
`/admin/users/{id}/subnet`.
