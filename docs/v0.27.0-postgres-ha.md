## Goal

Move the primary skygate + headscale deployment to a new VM
(`45.152.198.217`) with **PostgreSQL** as the database backend,
providing:

- **Reliability**: synchronous streaming replication, RPO = 0
- **Resilience**: auto-failover via Patroni, RTO = 30-60 s
- **Maintainability**: point-in-time recovery via WAL archiving to S3,
  real backups, no more "SQLite file on a Docker volume" footgun
- **Auditability**: PostgreSQL native logging of slow queries,
  locks, and connection state

The current VM (`192.168.13.69`, public IP `95.165.170.190`) becomes
a **warm standby** — same stack installed, DB replica + headscale
replica running, ready to take over in <1 min if the new VM dies.

---

## Why PostgreSQL (vs. Litestream + SQLite)

This was a deliberate architectural decision, not a default. The
operator chose PostgreSQL because:

1. **RTO < 1 min** (Litestream's 5-15 min failover is too slow for
   what we want — the operator's household tailnet, while not
   business-critical, deserves a snappy recovery)
2. **RPO = 0** (synchronous replication, not "we lost up to 5 sec
   of writes")
3. **Real DR tooling** (`pg_dump`, `pgBackRest`, WAL archiving to
   S3) instead of bespoke Litestream scripts
4. **Headroom for 100+ users** (the SQLite `skygate.db` is currently
   ~30 MB; the moment it crosses 1 GB, SQLite starts hurting on
   heavy writes; PostgreSQL scales linearly)
5. **Industry standard** (every operator who joins the project
   already knows PG; nobody needs to learn Litestream)
6. **Patroni is mature** (battle-tested at Gitlab, Zalando, etc.;
   failover automation is well-documented)

The cost: ~2-3 weeks of dev work to migrate the DB layer
(`?` → `$1`, `INSERT OR IGNORE` → `ON CONFLICT`, schema
type changes). This is a one-time investment.

### Roles

| Component | NEW VM (primary) | OLD VM (standby) |
|---|---|---|
| PostgreSQL | primary, accepts writes | replica, read-only |
| skygate | running, serves traffic | running, idle (not on LB) |
| headscale | running, serves tailnet | running, idle (not on LB) |
| Caddy | serving `headv2.skynas.ru` etc | serving nothing (DNS points elsewhere) |
| Tailscale | joined, `skygate-vm` identity | joined, same identity (if we move it) |
| etcd | node 1 | node 2 (we'll add a 3rd on a 3rd VM later) |
| S3 | writes (WAL archive) | writes (Litestream for headscale) |

### Failure modes

| Failure | Detection | Response | RTO |
|---|---|---|---|
| skygate dies | Docker restart (auto) | container back in 5-10 sec | 10 sec |
| headscale dies | Docker restart (auto) | container back in 5-10 sec | 10 sec |
| NEW VM kernel panic | etcd loses quorum, Patroni promotes replica | OLD VM takes over | 30-60 sec |
| NEW VM network partition | PG replica stops receiving WAL, Patroni triggers election | OLD VM becomes primary, DNS repointed | 30-60 sec + DNS TTL (5 min) |
| OLD VM dies | NEW VM keeps running, replica gone | no impact | 0 |
| S3 unavailable | WAL archive fails, but primary keeps accepting writes | data loss if NEW VM dies before WAL catches up | RPO = WAL archive lag |
| Both VMs die | manual recovery from S3 backup | restore from last `pg_dump` + headscale Litestream | 30 min |

### Phase 1 — Code: SQLite → PostgreSQL (Days 1-7)

This is the biggest single block of work. The DB layer touches
~30 files, ~5,000 lines of SQL-execution code. We do it in a
feature branch so main stays deployable throughout.

#### 1.1 — Driver + abstraction (Day 1, 4 hours)

- Add `github.com/jackc/pgx/v5` to `go.mod`
- Add `github.com/jackc/pgx/v5/stdlib` (for `database/sql` compatibility)
- New file: `internal/db/driver.go` — `Open(dsn string) (*sql.DB, error)`:
  - If `SKYGATE_DB_DSN` env var is set → use PG (`sql.Open("pgx", dsn)`)
  - Else → fall back to SQLite (existing behavior)
- `OpenForTest` stays SQLite (for fast unit tests)
- `OpenForIntegrationTest` (new) — opens PG with a temp schema per test

#### 1.2 — Schema conversion (Days 2-3, 1.5 days)

For each migration file (`migrations_v0.25.go` through
`migrations_v0.43.go`), produce a PG version. Key conversions:

| SQLite | PostgreSQL |
|---|---|
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGSERIAL PRIMARY KEY` (or `BIGINT GENERATED ALWAYS AS IDENTITY`) |
| `TEXT` | `TEXT` (no change) |
| `DATETIME` | `TIMESTAMP WITH TIME ZONE` |
| `INTEGER` (epoch seconds) | `BIGINT` (or `TIMESTAMPTZ` and let PG do conversion) |
| `BOOLEAN` (0/1) | `BOOLEAN` |
| `BLOB` | `BYTEA` |
| `INSERT OR IGNORE` | `INSERT ... ON CONFLICT DO NOTHING` |
| `INSERT OR REPLACE` | `INSERT ... ON CONFLICT (col) DO UPDATE SET ...` |
| `PRAGMA journal_mode=WAL` | (not needed in PG) |
| `PRAGMA foreign_keys=ON` | (PG defaults ON) |
| `?` placeholders | `$1, $2, $3` (numbered) |
| `?` in URL (DSN) | `?` in PG DSN is fine, but format is `postgres://user:pass@host/db?sslmode=disable` |
| `datetime('now')` | `now()` or `CURRENT_TIMESTAMP` |
| `strftime('%Y-%m-%dT%H:%M:%fZ', 'now')` | `to_char(now() AT TIME ZONE 'UTC', 'YYYY-MM-DD"T"HH24:MI:SS.MS"Z"')` |
| `COALESCE(x, 'default')` | same |
| `IFNULL(x, y)` | `COALESCE(x, y)` (same) |
| `json_extract(col, '$.field')` | `col->>'field'` or `col->'field'` (jsonb) |
| `GROUP_CONCAT(x)` | `string_agg(x, ',')` |
| `length(x)` | `length(x)` (same, but careful: PG counts characters, SQLite counts bytes for VARCHAR) |
| `substr(x, start, len)` | `substring(x from start for len)` |
| `strftime('%s', col)` | `EXTRACT(EPOCH FROM col)::bigint` |
| `datetime(col, 'unixepoch')` | `to_timestamp(col)` |

#### 1.3 — Query rewrite (Days 3-4, 1.5 days)

Mechanical conversion of all `?` → `$1, $2, ...` in:
- `internal/db/queries.go` (the main 34 KB file)
- `internal/db/portal_users.go` (16 KB)
- `internal/db/portal_users_controlplane.go` (8 KB)
- `internal/db/node_owner_map.go` (24 KB)
- `internal/db/device_rules.go` (19 KB)
- `internal/db/exit_servers.go`, `exit_node_health.go`, `acl_snapshots.go`
- `internal/db/telegram_*.go`
- `internal/db/audit_log.go`, `preauth_keys.go`, `secrets.go`, etc.
- `internal/db/migrations_v*.go` (also rewrite migrations as above)
- `internal/acl/acl.go` (uses SQL directly)
- `internal/handlers/*.go` (raw SQL in a few places)

This is largely a search-replace, but with care:
- `?` inside string literals is NOT a placeholder (false positives)
- `?` in `INSERT INTO ... VALUES (?, ?, ?)` → 3 numbered placeholders
- `?` in `WHERE x = ? AND y = ?` → 2 numbered placeholders
- Counter resets per query, not per file

#### 1.4 — Tests (Day 5, 0.5 day)

- `make test` runs both

#### 1.5 — docker-compose for local dev (Day 5, 0.5 day)

#### 1.6 — Migration script (Day 6, 0.5 day)

- `scripts/migrate_sqlite_to_pg.sh`:
  1. Dump current SQLite skygate.db as SQL (`sqlite3 .dump`)
  2. Rewrite SQL to PG-compatible (substitutions per the table above)
  3. Create fresh PG database
  4. Run all `migrations_v0.25..v0.43` to get the schema

### Phase 3 — VM migration: skygate + headscale (Days 12-15)

#### 3.1 — Tailscale state migration (Day 12, 1 hour)

skygate's tailscaled state (in `data/ts/`) is per-VM. If we move
the directory, the tailscale identity changes — which means a
new preauth is needed.

**Two options**:

**Option A: keep same tailscale identity (recommended)**:
- Use `tailscale down` on OLD VM (disconnects from tailnet)
- Use `tailscale logout` (clears state)
- Copy `data/ts/` from OLD VM to NEW VM (preserves nodekey)
- On NEW VM: `tailscaled & tailscale up --login-server=...`
- skygate-vm reappears with same identity, no client changes
- Tailscale clients see the same node, just at a different IP

**Option B: new identity, re-preauth (simpler, more disruptive)**:
- On NEW VM: `tailscale up --authkey=NEW_KEY`
- skygate-vm appears as a new node
- Old node gets auto-removed after expiry (or we delete it)
- More client-visible changes

We'll do Option A.

#### 3.6 — Subnet-router container (Day 13, 0.5 hour)

- `skyadmin-subnet-router` is a tailscale container on OLD VM
- It has its own tailscaled state
- We can either:
  - (A) Re-run on NEW VM (simpler, but the Tailscale node gets
    a new identity → tag:subnet-router is re-applied by skygate
    on next /my/devices load → all 4 users get pinged in audit)
  - (B) Keep on OLD VM (won't work once we repoint DNS — the
    sub-router advertises its LAN via the control server, but the

    still talk to the tailnet, but the routes it advertises
    come from the subnet it sees on its local network, which
    is OLD VM's network, not user's actual LAN)
  - **(C) Just re-run on NEW VM (option A). User can rerun
    `setup.sh` after DNS cutover to get a fresh sub-router on
    their actual LAN**

We go with (C) for simplicity. The pilot sub-router was for
testing; real users have their own LAN devices.

### Phase 4 — DR drill (Day 15, 0.5 day)

The whole point of HA is to test it works. We simulate a failure.

**Drill A: skygate process crash (sanity check)**:
- `docker stop skygate` on NEW VM
- Wait 5 sec
- `docker compose up -d skygate` — restarts
- `/healthz` returns 200
- Manual login works

**Drill B: headscale process crash**:
- `docker stop headscale` on NEW VM
- Wait 5 sec
- Restart
- Verify: 13 nodes, Tailscale clients reconnect within 60s

**Drill C: PG primary dies (real HA test)**:
- `pg_ctl stop` on NEW VM (or `systemctl stop postgresql`)
- Wait 60 sec
- On OLD VM: `patronictl list` — OLD should be Leader
- Start skygate on OLD VM with new DSN (pointing at OLD VM's PG)
- Wait for /healthz=200
- Operator logs in at `http://192.168.13.69:8080` (OLD VM's port)
- All 4 users' data is intact
- Headscale on OLD VM is serving the tailnet (same noise key)
- Time: measure RTO end-to-end

**Drill D: NEW VM kernel panic (simulated)**:
- `echo c > /proc/sysrq-trigger` (or just `docker stop` everything)
- OLD VM must take over
- Same as Drill C, plus the Caddy failover

**Drill E: split brain (network partition)**:
- `iptables -A INPUT -s 192.168.13.69 -j DROP` (block traffic from OLD VM)
- NEW VM: PG primary, no change
- OLD VM: loses connection to NEW, but keeps running
- After 30 sec, restore: `iptables -D INPUT -s 192.168.13.69 -j DROP`
- Verify: OLD VM rejoins as replica, no data loss

**Drill F: S3 unavailable (WAL archive fails)**:
- `iptables -A OUTPUT -d s3.amazonaws.com -j DROP`
- Primary keeps accepting writes (queueing WAL locally)
- Restore network — WAL catches up
- `wal-g backup-list` shows new backup

**Phase 4 done when**: all 6 drills pass, RTO ≤ 60 sec, RPO = 0.

## Rollback plan

If at any point during the migration things go wrong:

1. **DNS rollback** (operator): change `head.skynas.ru` A record back to
   `95.165.170.190` (OLD VM). TTL is 5 min, so within 5 min everything
   is on OLD VM again.
2. **Code rollback**: `git revert` the postgres-migration commits. Main
   branch is on SQLite, OLD VM runs the old code.
3. **DB rollback** (if data corruption): we keep a SQLite dump of the
   pre-migration state for 30 days. Restore via `deploy/restore.sh`.
4. **Worst case**: rebuild OLD VM from `deploy/backup.sh` artifacts
   (15-30 min per `docs/disaster-recovery.md`).

## Decisions log

| Decision | Choice | Rationale |
|---|---|---|
| PG vs. Litestream | **PG** | Operator's strategic call: RTO < 1 min, RPO = 0, real DR tooling |
| Self-hosted vs. managed | **Self-hosted** | Cheaper, more control, PG isn't that hard |
| Patroni vs. pg_auto_failover | **Patroni** | Industry standard, more docs, easier to find help |
| 1 vs. 2 replicas | **1** | 4 users today, 1 replica covers 99% of failures |
| headscale DB | **SQLite + Litestream** | headscale 0.29.x has no PG; 0.30+ does but not stable yet |
| New skygate hostname | **`skygate.skynas.ru`** | Replaces broken `gate.skynas.ru` |
| Per-user headscale (v0.23.0) | **Stays** | Procedurally same, runs on NEW VM |
| Subnet-router container | **Re-run on NEW VM** | Simple, just a new Tailscale identity (handled by sub-router flow) |
| skygate DB driver | **pgx/v5** | Modern, fast, well-maintained (not lib/pq) |
| Migration tool | **Custom (existing pattern)** | skygate has its own `migrations_v0.XX.go` — keep that pattern |
| DB schema diffing | **Per-version (v0.25-v0.43)** | 19 migration files, 19 PG versions |
| Session storage | **JWT (stateless)** | Already stateless, no change needed for HA |
| WAL archiving tool | **wal-g** | S3-aware, popular, point-in-time recovery |
| HAProxy | **Yes** | pg-aware, health-checks Patroni REST API |
| DNS testing | **`headv2.skynas.ru`, `skygatev2.skynas.ru`** | v2 suffix, removed after cutover |

## Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Migration breaks audit log | Medium | High | Row-count verification + manual audit query per user |
| PG migration introduces performance regression | Low | Medium | Load test with 100 concurrent before cutover |
| Patroni fails over spuriously | Low | High | Tune `ttl` (30s) and `maximum_lag_on_failover` (1MB) |
| DNS propagation takes longer than 5 min | Low | Low | Use `dig +short` to monitor; have old VM as backup |
| WAL archive to S3 fails silently | Low | High | Monitor `wal-g backup-list` daily; alert via Telegram |
| headscale 0.30+ lands and we want to migrate to PG | Low | Low | Blocked, will re-evaluate when 0.30 ships |
| Both VMs go down (correlated failure) | Very low | High | DR runbook: restore from S3 backup |
