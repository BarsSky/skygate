# v0.28.0 — VM Provisioning UI

**Status**: planned (post-v0.27.0)
**Target release**: v0.28.0
**Author**: Mavis (skygate)
**Operator**: skyadmin

---

## Goal

Replace the manual "open a shell, ssh into the new VM, copy-paste
the 30-step deployment script" flow with a single-button
**Provision new VM** UI inside skygate's admin section.

The operator fills a form (hostname, IP, SSH key, role, components),
clicks Start, and skygate runs the entire deployment as a
state-machine-driven background job — with progress visible in
the web UI, error reporting per step, and audit-log entries for
every action.

This is the v0.27.0 HA story made *operable*. v0.27.0 ships
the *architecture* (PG primary + replica, Patroni failover,
WAL archive). v0.28.0 ships the *operator workflow* for adding
nodes to that architecture without a 90-minute manual runbook.

## Why now

- v0.27.0 establishes the PostgreSQL HA baseline (Phase 2.1-2.5)
  on a single primary + replica pair. The next node addition
  (DR replica in a third AZ? Cloud-burst replica for batch jobs?
  Operator's home-lab machine for a fourth user?) repeats the
  same 30-step pattern.
- Manual runs of 30+ steps are where copy-paste errors live.
  A provisioner that runs the same code on every new node
  guarantees consistency.
- The state machine + audit log + idempotent steps are all
  skygate-shaped concerns (in-DB state, encrypted secrets,
  per-step logs). The SSH-and-bash part is plumbing.

## Out of scope (v0.28.0)

- **Multi-cloud provisioning** (AWS, GCP, YC, Azure APIs) — the
  v0.28.0 input is "I have a VM, ssh into it". Cloud-vendor
  APIs to *create* the VM are a v0.30.0+ feature.
- **Auto-scaling** (spin up a new node when N fail) — v0.31.0+
  (depends on v0.30.0 metrics + alerting).
- **Decommissioning** (cleanly remove a node from the cluster) —
  v0.29.0 (after we have a proven provision path).

## Backend: state machine

The provisioner is a sequential state machine. Each step is a
Go function that:

1. Receives the `*Job` (with all the input + secrets)
2. Logs its start to `provision_step_logs`
3. Runs the actual work (mostly: SSH + bash, sometimes: skygate API)
4. Updates the job's `current_step` field
5. Returns `nil` (move to next step) or `error` (mark job as
   `failed:<step>`, return)

Steps are listed in `provision/steps.go` as a slice:
```go
var defaultSteps = []Step{
    sshCheckStep,
    installDockerStep,
    installTailscaleStep,
    installPostgresStep,
    configureReplicationStep,
    pgBasebackupStep,
    startPatroniStep,
    verifyStreamingStep,
    installHeadscaleStep,
    installSkygateStep,
    finalVerifyStep,
}
```

Each step function is responsible for being **idempotent** —
running it twice has the same effect as running it once. This
is what makes the Retry button safe.

The state machine runs in a background goroutine kicked off
by `POST /admin/cluster/provision`. The HTTP request returns
201 with the job ID; the operator polls
`GET /admin/cluster/provision/<id>` for live status (rendered
via SSE or short-poll every 2s).

## Step details

For each step, the implementation pattern is the same:
1. Build a small Go function that runs `ssh.Run("bash -c '<cmd>'")`
2. Capture stdout/stderr (capped at 64 KB per step to avoid
   blowing up the DB)
3. Check the exit code; if non-zero, return an error with the
   last 50 lines of stderr
4. Log start + end timestamps to `provision_step_logs`

The steps in v0.28.0:

### 1. `sshCheckStep`
```bash
ssh -i <key> -o StrictHostKeyChecking=accept-new \
    root@<host> 'whoami && uname -m && cat /etc/os-release | head -1'
```
Verifies: SSH key works, OS is Linux/amd64. The `accept-new`
auto-trusts the host key on first connect (subsequent connects
verify against the stored fingerprint).

### 2. `installDockerStep`
Idempotent: skips if `docker --version` already shows a recent
version. Otherwise runs:
```bash
install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
    | gpg --dearmor -o /etc/apt/keyrings/docker.gpg
echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.gpg] \
    https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" \
    > /etc/apt/sources.list.d/docker.list
apt-get update
DEBIAN_FRONTEND=noninteractive apt-get install -y \
    docker-ce docker-ce-cli containerd.io docker-buildx-plugin \
    docker-compose-plugin
```

### 3. `installTailscaleStep`
Joins with a per-job ephemeral auth key generated server-side:
```go
authKey, _ := generatePreauthKey(brandNew=true, ephemeral=true, singleUse=true)
ssh.Run(fmt.Sprintf(`tailscale up --login-server=%s --authkey=%s \
    --hostname=%s --accept-routes`, headscaleURL, authKey, hostname))
```
The auth key is stored on the job so the operator can audit
which node joined with which key. (No reusable keys; the job
exits with `failed: tailscale_join` if the join doesn't complete
within 60s.)

### 4. `installPostgresStep`
Same script as v0.27.0 Phase 1.3.0 — installs PG 16 via PGDG
repo, creates the cluster, sets up a 2 GB swap file (the
1.9 GB VPS case from the v0.27.0 runbook).

### 5. `configureReplicationStep` (replica only)
Edits `postgresql.conf` and `pg_hba.conf`:
```
wal_level = replica
max_wal_senders = 5
wal_keep_size = 1GB
synchronous_standby_names = 'skygate_pg_replica'  # or '*' for sync-replication
hot_standby = on
```
And adds a `host replication skygate_repl <primary>/32 scram-sha-256`
line to pg_hba.conf. Restart the cluster.

### 6. `pgBasebackupStep` (replica only)
```
sudo -u postgres pg_basebackup -h <primary> -U skygate_repl \
    -D /var/lib/postgresql/16/main -P -Xs -R
```
The `-R` flag writes `standby.signal` + `primary_conninfo` so
the replica starts streaming on first run.

### 7. `startPatroniStep` (replica only)
Installs Patroni via pip, writes `/etc/patroni.yml` pointing
at the existing etcd cluster (from v0.27.0 Phase 2.5), starts
the systemd unit. Verifies with `patronictl list` that the
new node shows as `Replica` with `Lag: 0`.

### 8. `verifyStreamingStep` (replica only)
On the **primary**, runs:
```sql
SELECT client_addr, state, sent_lsn, replay_lsn,
       (sent_lsn - replay_lsn) AS lag_bytes
  FROM pg_stat_replication
 WHERE application_name = '<new-hostname>';
```
Expects: `state = 'streaming'`, `lag_bytes = 0` (or near-zero
after the initial catchup). If state is `startup` for more than
5 min, the step fails and the operator gets a Telegram alert.

### 9. `installHeadscaleStep` (optional, ticked in form)
If the new VM is also going to run a headscale container
(e.g. disaster-recovery node), rsync the headscale data:
```bash
rsync -a --delete \
    /home/skyadmin/headscale/{config,data,headplane,users}/ \
    root@<new-vm>:/home/skyadmin/headscale/{config,data,headplane,users}/
```
Critical: the `noise_private.key` (in `headscale_data`) is what
makes the replica the **same** control-server identity. If the
file is missing, the replica becomes a new control server and
all Tailscale clients need to re-auth.

### 10. `installSkygateStep` (optional)
```bash
rsync -a /home/skyadmin/skygate/ root@<new-vm>:/home/skygate-admin/skygate/
ssh root@<new-vm> 'cd /home/skyadmin/skygate && \
    docker compose build skygate && docker compose up -d skygate'
```
The `.env` is rewritten to point at the local PG (`SKYGATE_DB=postgres://localhost:5432/skygate`).
Caddy + tailnet join + admin bootstrap are the same as a fresh
deploy.

### 11. `finalVerifyStep`
```bash
curl -fsS http://<new-vm>:8080/healthz     # 200
curl -fsS http://<new-vm>:8080/readyz      # {"healthy":true, "db":"ok", ...}
```
If both pass, the job is marked `done` and a Telegram message
goes to the configured chat:
```
✅ Provision job #42 completed (svyatoslava2)
   - PG replica streaming
   - Headscale joined as node 27
   - skygate: /healthz=200, /readyz=ok
   - Time: 8m 32s
Next steps:
   - DNS: point head.skynas.ru to <new-vm> (or load-balance)
   - DR test: docker stop postgresql on primary
```

## Storage

New tables (migration v0.44):

```sql
-- One row per provisioning attempt
CREATE TABLE provision_jobs (
    id BIGSERIAL PRIMARY KEY,
    hostname TEXT NOT NULL,
    ssh_address TEXT NOT NULL,
    ssh_user TEXT NOT NULL,
    ssh_key_enc BYTEA NOT NULL,           -- AES-GCM with SKYGATE_SECRET_KEY
    role TEXT NOT NULL,                   -- 'replica' | 'primary'
    primary_host TEXT,
    primary_port INT,
    primary_user TEXT,
    primary_pass_enc BYTEA,               -- AES-GCM
    components JSONB NOT NULL,            -- {pg:true, tailscale:true, headscale:true, skygate:true}
    telegram_chat_id BIGINT,
    status TEXT NOT NULL DEFAULT 'pending',  -- pending|running|done|failed:<step>
    current_step TEXT,                    -- step currently running
    started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    finished_at TIMESTAMPTZ,
    error TEXT
);

-- One row per (job, step) — for the per-step log timeline
CREATE TABLE provision_step_logs (
    id BIGSERIAL PRIMARY KEY,
    job_id BIGINT NOT NULL REFERENCES provision_jobs(id) ON DELETE CASCADE,
    step TEXT NOT NULL,
    started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    finished_at TIMESTAMPTZ,
    exit_code INT,
    stdout_tail TEXT,                     -- last 64 KB
    stderr_tail TEXT,
    CONSTRAINT fk_provision_step_logs_job FOREIGN KEY (job_id)
        REFERENCES provision_jobs(id) ON DELETE CASCADE
);
```

Indexes:
- `provision_jobs(status, started_at)` — list recent jobs
- `provision_step_logs(job_id, started_at)` — per-job timeline

## Security

- **Admin only**: the UI lives under `/admin/cluster/` and the
  middleware already requires `is_admin=1` for everything under
  `/admin/`. No change to the auth stack.
- **Encrypted secrets**: `ssh_key` and `primary_pass` are
  AES-GCM-encrypted with the existing `SKYGATE_SECRET_KEY`
  (32 bytes hex). Stored as `BYTEA`. Decrypted only when the
  step is actively running; never logged.
- **No password in plain text anywhere**: the form input is
  POSTed over HTTPS, decrypted in-memory for the step, and
  re-encrypted before DB storage.
- **Audit log**: every job event goes through
  `db.AppendAuditLog` with action `provision_*` so the
  per-user audit page shows who triggered what.
- **Rate limit**: max 5 jobs per 24 h per admin. Prevents
  accidental button-mash from spinning up 50 half-configured
  replicas.
- **Cleanup**: secrets are stored for 7 days post-completion
  (so the operator can re-run a failed step), then a daily
  job (`scripts/cleanup_old_provision_secrets.sh`) zeroes the
  `ssh_key_enc` and `primary_pass_enc` columns.

## Open questions

1. **Where do the SSH private keys come from?** The form has
   a paste field. We could also offer "use a pre-uploaded
   skygate-managed key" (operator uploads to `/admin/keys`,
   skygate fingerprints and reuses). v0.28.0 starts with the
   paste field; the managed-keys flow is v0.29.0 polish.

2. **What about Windows VMs?** The current plan is Linux-only.
   The 11 steps would change substantially for Windows
   (PowerShell + WinRM instead of bash + SSH). Out of scope
   until someone asks.

3. **What if the new VM is on a different network?** Currently
   the steps assume the new VM can reach the primary's
   `HEADSCALE_URL` and the primary can reach the new VM's
   `PG port 5432`. v0.28.0 doesn't open firewalls; the
   operator must do that out-of-band. v0.30.0+ could
   integrate with cloud security-group APIs.

4. **What if the new VM has a different arch (e.g. arm64)?**
   v0.27.0's PG primary is `amd64`. The install_docker and
   install_postgres steps detect arch via `dpkg --print-architecture`
   and use the right `*.deb`. arm64 not tested in v0.28.0.

5. **Idempotency in the face of partial failure.** A
   provision job that fails at step 7 with 6 steps done
   has the new VM in a half-configured state. Retry resumes
   from step 7 (idempotent) but the operator needs to be
   told the VM is in a "partially-provisioned" state until
   the job is `done`. The UI shows this in red; the next
   provision attempt on the same hostname warns ("VM svyatoslava2
   is partially provisioned, retry the existing job?").
