Routing management · API Tokens · Exit Rules

For users — how to add rules

  1. Go to /my/exit-rules.
  2. In the "Add rule" form:
    • Device — pick your device (e.g. "my-laptop").
    • Exit node — pick a server (e.g. "node-europe").
    • Action — Allow (pass traffic) or Deny (drop traffic).
    • Target type — IP address, subnet (10.0.0.0/24), or domain (example.com).
    • Value — IP/subnet/domain.
  3. Click "Add". The rule is applied immediately.
  4. On the device: tailscale up --accept-routes (Windows/Linux) or tailscale up --exit-node=node-name (Android/iOS).

DNS auto-update (since July 2026)

For rules with target type "domain", Skygate every 5 minutes automatically:

  1. Resolves the domain via DNS (if it doesn't resolve yet, adds as target_type=domain and retries later).
  2. Finds static subdomains (e.g. static.rutracker.cc for rutracker.org) and adds their IPs.
  3. Updates /32 rules: removes stale IPs, adds new ones.
  4. Syncs advertised-routes on exit-nodes and approves them through headscale nodes approve-routes.

In the rules table domain rules are marked with the auto badge.

All IPs added by the auto-updater have parent_domain in the DB — autoupdater distinguishes them and doesn't touch manual /32 rules.

⚠️ Some sites (rutracker.org, kinozal.tv, etc.) use a separate subdomain for static content. If autoupdater finds it in HTML, it auto-resolves and adds it. If not — file a request to the admin, we'll add to the knownSubdomains map.

Known subdomains: rutracker.org → static.rutracker.cc

Rule deduplication

When you add a rule Skygate checks uniqueness by the combination (device_id, exit_node, target_type, target_value):

  • ✅ New rule — added, you'll see the green "Rule added" alert.
  • 🔁 Duplicate — the rule already exists, you'll see the red alert "A rule for X already exists".
  • ⚠️ Partial add — some IPs already exist (for domains resolving to shared Cloudflare IPs), you'll see the yellow alert.
  • ⚠️ DNS warning — domain doesn't resolve, saved as target_type=domain, autoupdater will try on the next cycle.

API Tokens — access tokens

For API access use a personal token. It's safer than a cookie session and more convenient for AI assistants and scripts.

  1. Go to /my/tokens → "Create token", specify a label (e.g. "Hermes").
  2. Copy the token — it's shown only once. Store it safely.
  3. Use in the header: Authorization: Bearer <your token>
  4. The token has no expiration. Revoke anytime on /my/tokens.
# With token (recommended):
curl -H "Authorization: Bearer <token>" \
  {{.BaseURL}}/my/exit-rules/api

# Bulk create:
curl -X POST -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"rules":[{"device_id":5,"exit_node":"node-europe","target_type":"subnet","target_value":"10.0.0.0/24","action":"accept"}]}' \
  {{.BaseURL}}/my/exit-rules/api

How it works

Skygate uses advertised routes for selective routing (split-tunnel).

  1. Exit nodes (VPS servers) advertise whitelist subnets via --advertise-routes.
  2. Headscale (control plane) approves and distributes routes to clients per ACL — each device gets only its own subnets.
  3. Clients:
    • Windows/Linux: tailscale up --accept-routes — split-tunnel (only whitelist → exit-node)
    • Android/iOS: tailscale up --exit-node=X — full tunnel
  4. When rules change Skygate auto-updates advertised-routes on all exit-nodes + autoupdater watches domains.

Tailnet — private mesh network

Tailnet is a mesh network based on Tailscale/headscale. All devices in the tailnet get IPs in the 100.64.0.0/10 range and see each other directly, regardless of physical location (home, office, datacenter, behind CGNAT).

Example:

my-phone (100.64.0.5) ──► exit-node (100.64.0.3) ──► internet (8.8.8.8)

When a client uses an exit-node, its internet traffic goes through that exit-node's VPS. For external sites the request source will be the VPS IP, not the client's.

⚠️ Cloudflare Challenge: some sites (rutracker.org, etc.) show a CAPTCHA when accessed from an exit-node IP outside Russia. Fix: use a Russian exit-node (sharlotta) for such sites, or work via --exit-node=karolina for all traffic.

Setting up an exit node (admin)

Run on each exit node (the VPS server that will be the egress point):

# Script: /static/scripts/skygate_exit_node_setup.sh
# Or manually:

tailscale set \
  --ssh \
  --advertise-exit-node \
  --advertise-routes="10.0.0.0/24,172.16.0.0/16,192.168.0.0/24,..."

⚠️ --advertise-routes replaces the list. 0.0.0.0/0 and ::/0 are kept automatically.

Rollback: tailscale set --advertise-exit-node --advertise-routes="0.0.0.0/0,::/0"

Requirements for an exit node: ip_forward=1, iptables MASQUERADE (auto-configured with --advertise-exit-node).

DERP relay

DERP (Designated Encrypted Relay for Packets) is a relay server that carries traffic when a direct connection between clients is impossible (CGNAT, firewall, NAT).

┌──────────┐   TLS 1.3 + WSS    ┌─────────┐   WebSocket   ┌──────────┐
│ my-phone │ ──────────────────▶│  DERP   │──────────────▶│ exit-vps │
│ 4G CGNAT │   TCP 443 (HTTPS)  │  relay  │  TCP 443      │  VPS EU  │
└──────────┘                    └─────────┘               └──────────┘
                                      │
                          sees: client IPs, packet sizes
                          does NOT see: contents (TLS 1.3 + WireGuard)

When it's used:

  • 🟢 Direct — clients connect directly (LAN, NAT punched). Maximum speed.
  • 🟡 DERP relay — direct not possible. The nearest relay is auto-picked. Slower, but always works.

DERP uses TLS 1.3 on TCP 443 — traffic looks like regular HTTPS, including to DPI.

Bulk delete (multi-delete)

The rules table has checkboxes and a sticky bar at top and bottom:

  • Select all in top and bottom bars — selects all visible rules.
  • Type filter (All types / Domains only / /32 only / IP only) — narrows the selection.
  • Search by IP/subnet/domain — hides non-matching rows.
  • Counter "Selected: N" — updates in real time in both bars.
  • Button "Delete selected" at the bottom — cascade-deletes all selected rules.

Cascade delete: when you delete a rule with type domain, all related /32 rules with the same parent_domain are removed automatically. Without this, autoupdater would leave "orphaned" /32 for domains the user no longer tracks.

Single delete (the button in a row) is also cascade for domains.

Rule counters

Counters (per-user / per-device / system) count only user-set ("user-facing") rules:

  • Counted: target_type=ip, target_type=subnet without parent_domain, target_type=domain.
  • NOT counted: target_type=subnet with parent_domain (these are /32 created by autoupdater for domain resolution).

In other words: if you have 9 domains and autoupdater added 240 /32 IPs, the per-user counter (default 200) does not count the auto-expanded IPs. Each domain counts as 1 rule.

Settings: SKYGATE_MAX_RULES_PER_DEVICE (default 200), SKYGATE_MAX_TOTAL_RULES (default 10000), SKYGATE_USER_MAX_RULES (optional, format user1:500,user2:1000).

Example: 9 domains + 130 manual /32 + 240 autoupdater /32 = 379 rules total, but the counter sees only 9+130=139, the other 240 are service (auto-resolved).

Cleanup orphaned /32 (admin)

If you used to delete rules manually and "orphaned" /32 remain (autoupdater can't remove them since the parent domain is gone), open /admin/exit-rules/cleanup. There:

  • Analysis of current state: distinct device_id, distinct hostname, pending /32 deletions, pending backfill device_ip.
  • Button "Apply" — merges duplicate device_id under one canonical (by hostname) and backfills device_ip for all rules.
  • Idempotent: re-running does nothing if the data is already clean.

Visibility in the tailnet (ACL)

Skygate generates a per-user ACL policy. Each user sees and reaches only their own devices plus shared exit-nodes (tag:public / tag:exit-node).

{
  "acls": [
    {"src": ["skyadmin@tsnet.skynas.ru"], "dst": ["skyadmin@tsnet.skynas.ru:*"]},
    ... per portal user ...
    {"src": ["*"], "dst": ["tag:public:*"]},
    {"src": ["*"], "dst": ["tag:exit-node:*"]},
    {"src": ["*"], "dst": ["*:*"]}
  ],
  "tagOwners": {
    "tag:private": [skyadmin, michail, guest, daniil, ...],
    "tag:public": ["skyadmin@tsnet.skynas.ru"],
    "tag:exit-node": ["skyadmin@tsnet.skynas.ru"]
  }
}

First match wins — if *:* were first, it would open the entire tailnet to everyone. Right now the first rule is only your own; Tailscale Android shows only your devices plus exit-nodes. Other users' tag:private are not visible to each other.

Auto-tagging (tag:private)

When you add a new device through Skygate it auto-receives the tag:private tag in headscale. This happens through backfillNodeOwnership on the first load of /my/devices:

  • If a node was registered through a skygate-issued preauth key, it is auto-tagged tag:private.
  • Exit-nodes (karolina, emilia, sharlotta) stay tag:public — the admin tags them manually through /admin/devices.
  • Tag changes apply to headscale through the CLI fallback (docker exec headscale headscale nodes tag). Visibility on Tailscale Android updates within ~60s.

To set a tag manually (e.g. to share a device) — the admin does it through /admin/devices.

Common pitfalls

  • target_type=ip without /32 — Skygate auto-adds /32 (e.g. 5.5.5.5 → 5.5.5.5/32) so headscale approve-routes doesn't fail with parsing route: no '/'.
  • Shared IP for two domains (e.g. gcr.io and ghcr.io sometimes resolve to one IP) — Skygate stores one /32 row, no duplicates. Autoupdater won't remove this IP while any of the domains needs it.
  • DNS doesn't resolve when adding a domain — the rule is saved as target_type=domain, autoupdater will try on the next cycle (up to 5 minutes).
  • Cloudflare Challenge on Russian sites (rutracker etc.) — use a Russian exit-node (sharlotta), not one abroad.

IP-range sources

Many services publish current IP ranges. Examples:

  • Service A: https://example.com/ips.txt
  • Service B: https://example.com/ipranges.json
  • Provider C: https://example.com/ips-v4

IP ranges change over time. Recommended:

  • For subnets — update every 1-3 months manually via AI.
  • For domains — autoupdater updates IPs automatically, nothing to do.
