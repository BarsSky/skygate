Exit Rules — справка

Управление маршрутизацией · API Tokens · Exit Rules

Для пользователей — как добавить правила

  1. Перейдите на страницу /my/exit-rules.
  2. В форме «Добавить правило»:
    • Устройство — выберите своё устройство (например «my-laptop»).
    • Exit node — выберите сервер (например «node-europe»).
    • Действие — Allow (разрешить) или Deny (запретить).
    • Тип цели — IP адрес, подсеть (10.0.0.0/24), или домен (example.com).
    • Значение — IP/подсеть/домен.
  3. Нажмите «Добавить». Правило сразу применяется.
  4. На устройстве: tailscale up --accept-routes (Windows/Linux) или tailscale up --exit-node=node-name (Android/iOS).

DNS авто-обновление (с июля 2026)

Для правил с типом «домен» Skygate каждые 5 минут автоматически:

  1. Резолвит домен через DNS (если ещё не резолвится — добавляет как target_type=domain и попробует позже).
  2. Находит все статические поддомены (например static.rutracker.cc для rutracker.org) и добавляет их IP.
  3. Обновляет /32 правила: удаляет устаревшие IP, добавляет новые.
  4. Синхронизирует advertised-routes на exit-узлах и одобряет их через headscale nodes approve-routes.

В таблице правил доменные правила помечены бейджем auto.

Все IP, добавленные авто-апдейтером, имеют parent_domain в БД — autoupdater их различает и не трогает ручные /32 правила.

⚠️ Некоторые сайты (rutracker.org, kinozal.tv и др.) используют отдельный поддомен для статики. Если autoupdater найдёт его в HTML — он автоматически резолвит и добавит. Если нет — пришлите заявку админу, добавим в knownSubdomains map.

Известные поддомены: rutracker.org → static.rutracker.cc

Защита от дублирования

При добавлении правила Skygate проверяет уникальность по комбинации (device_id, exit_node, target_type, target_value):

  • ✅ Новое правило — добавлено, появится зелёный alert «Правило добавлено».
  • 🔁 Дубликат — правило уже существует, появится красный alert «Правило для X уже существует».
  • ⚠️ Частичное добавление — некоторые IP уже есть (для доменов, которые резолвятся в общие Cloudflare IP), появится жёлтый alert.
  • ⚠️ DNS warning — домен не резолвится, сохранён как target_type=domain, autoupdater попробует при следующем цикле.

API Tokens — токены доступа

Для доступа к API используйте персональный токен. Это безопаснее cookie-сессии и удобнее для AI-помощников и скриптов.

  1. Перейдите на /my/tokens → «Создать токен», укажите метку (например «Hermes»).
  2. Скопируйте токен — он показывается только один раз. Храните в безопасном месте.
  3. Используйте в заголовке: Authorization: Bearer <ва.../li>
  4. Токен действует без ограничения срока. Отозвать можно в любой момент на /my/tokens.
# С токеном (рекомендуется):
curl -H "Authorization: Bearer *** \
  {{.BaseURL}}/my/exit-rules/api

# Массовое создание:
curl -X POST -H "Authorization: Bearer *** \
  -H "Content-Type: application/json" \
  -d '{"rules":[{"device_id":5,"exit_node":"node-europe","target_type":"subnet","target_value":"10.0.0.0/24","action":"accept"}]}' \
  {{.BaseURL}}/my/exit-rules/api

Как это работает

Skygate использует advertised routes для избирательной маршрутизации (split-tunnel).

  1. Exit-узлы (VPS-серверы) анонсируют whitelist-подсети через --advertise-routes.
  2. Headscale (control plane) одобряет и раздаёт маршруты клиентам по ACL — каждое устройство получает только свои подсети.
  3. Клиенты:
    • Windows/Linux: tailscale up --accept-routes — split-туннель (только whitelist → exit-node)
    • Android/iOS: tailscale up --exit-node=X — полный туннель
  4. При изменении правил Skygate автоматически обновляет advertised-routes на всех exit-узлах + autoupdater следит за доменами.

Tailnet — частная mesh-сеть

Tailnet — это mesh-сеть на базе Tailscale/headscale. Все устройства в tailnet получают IP из диапазона 100.64.0.0/10 и видят друг друга напрямую, независимо от физического расположения (дома, в офисе, в дата-центре, за CGNAT).

Пример:

my-phone (100.64.0.5) ──► exit-node (100.64.0.3) ──► internet (8.8.8.8)

Когда клиент использует exit-node, его трафик в интернет идёт через VPS этого exit-узла. Для внешних сайтов источником запроса будет IP VPS-сервера, а не клиента.

⚠️ Cloudflare Challenge: некоторые сайты (rutracker.org и др.) показывают CAPTCHA если заходят с IP exit-узла вне РФ. Решение: используйте exit-узел в РФ (sharlotta) для таких сайтов, или работайте через --exit-node=karolina для всего трафика.

Настройка exit-node (администратор)

Выполнить на каждом exit-узле (VPS-сервер, который будет выходной точкой):

# Скрипт: /static/scripts/skygate_exit_node_setup.sh
# Или вручную:

tailscale set \
  --ssh \
  --advertise-exit-node \
  --advertise-routes="10.0.0.0/24,172.16.0.0/16,192.168.0.0/24,..."

⚠️ --advertise-routes заменяет список. 0.0.0.0/0 и ::/0 сохраняются автоматически.

Откат: tailscale set --advertise-exit-node --advertise-routes="0.0.0.0/0,::/0"

Требования к exit-node: ip_forward=1, iptables MASQUERADE (настраивается автоматически при --advertise-exit-node).

DERP relay

DERP (Designated Encrypted Relay for Packets) — relay-сервер, через который идёт трафик когда прямое соединение между клиентами невозможно (CGNAT, файрвол, NAT).

┌──────────┐   TLS 1.3 + WSS    ┌─────────┐   WebSocket   ┌──────────┐
│ my-phone │ ──────────────────▶│  DERP   │──────────────▶│ exit-vps │
│ 4G CGNAT │   TCP 443 (HTTPS)  │  relay  │  TCP 443      │  VPS EU  │
└──────────┘                    └─────────┘               └──────────┘
                                      │
                            видит: IP клиентов, размер пакетов
                            НЕ видит: содержимое (TLS 1.3 + WireGuard)

Когда используется:

  • 🟢 Direct — клиенты соединяются напрямую (LAN, пробили NAT). Максимальная скорость.
  • 🟡 DERP relay — direct невозможен. Автоматически выбирается ближайший relay. Медленнее, но работает всегда.

DERP устойчив к блокировкам: TLS 1.3 + TCP 443 выглядит как обычный HTTPS, DPI не видит содержимое.

API Reference

GET /my/exit-rules/api — список правил

# С токеном:
curl -H "Authorization: Bearer *** {{.BaseURL}}/my/exit-rules/api

# Или с cookie:
curl -b skygate_session=TOKEN {{.BaseURL}}/my/exit-rules/api

Ответ: {"rules": [{"id":1, "device_id":5, "device_name":"my-laptop", "exit_node":"node-europe", "target_type":"ip", "target_value":"1.2.3.4", "action":"accept", "device_ip":"100.64.0.x", "parent_domain":""}]}

Поле parent_domain — для /32 правил, созданных autoupdater'ом, содержит исходный домен.

POST /my/exit-rules/api — массовое создание

# С токеном:
curl -X POST -H "Authorization: Bearer *** \
  -H "Content-Type: application/json" \
  -d '{"rules":[
    {"device_id":5, "exit_node":"node-europe", "target_type":"subnet", "target_value":"10.0.0.0/24", "action":"accept"},
    {"device_id":5, "exit_node":"node-europe", "target_type":"ip",     "target_value":"1.2.3.4",     "action":"accept"},
    {"device_id":5, "exit_node":"node-europe", "target_type":"domain",  "target_value":"rutracker.org", "action":"accept"}
  ]}' \
  {{.BaseURL}}/my/exit-rules/api

# Или с cookie:
curl -X POST -b skygate_session=TOKEN \
  -H "Content-Type: application/json" \
  -d '{"rules":[...]}' \
  {{.BaseURL}}/my/exit-rules/api

Ответ: {"added": 3, "duplicates": 0, "errors": []}

Для target_type=domain сервер автоматически резолвит домен в IP (если DNS отвечает) и сохраняет каждую запись как subnet /32. Доменное правило само остаётся в БД для autoupdater.

Массовое удаление (multi-delete)

В таблице правил появились чекбоксы и sticky-панель вверху и внизу страницы:

  • Select all в верхней и нижней панели - выделяет все видимые правила.
  • Фильтр по типу (Все типы / Только домены / Только /32 / Только IP) - сужает выбор.
  • Поиск по IP/подсети/домену - скрывает неподходящие строки.
  • Счётчик «Выбрано: N» - обновляется в обеих панелях в реальном времени.
  • Кнопка «Удалить выбранные» внизу - каскадно удаляет все выделенные правила.

Каскадное удаление (cascade): при удалении правила с типом domain автоматически удаляются все связанные /32 правила с тем же parent_domain. Без этого autoupdater оставлял бы «осиротевшие» /32 для доменов, которые пользователь больше не отслеживает.

Одиночное удаление (кнопка в строке) тоже работает каскадно для доменов.

Лимиты правил

Лимиты (per-user / per-device / system) считают только «user-facing» правила:

  • Считаются: target_type=ip, target_type=subnet без parent_domain, target_type=domain.
  • НЕ считаются: target_type=subnet с parent_domain (это /32, созданные autoupdater'ом для резолва доменов).

Это значит: если у вас 9 доменов и autoupdater добавил 240 /32 IP, лимит per-user (по умолчанию 200) не блокирует добавление новых доменов. Домен считается как 1 правило.

Настройка: SKYGATE_MAX_RULES_PER_DEVICE (default 200), SKYGATE_MAX_TOTAL_RULES (default 10000), SKYGATE_USER_MAX_RULES (опционально, формат user1:500,user2:1000).

Пример: 9 доменов + 130 ручных /32 + 240 autoupdater /32 = 379 правил всего, но лимит видит только 9+130=139, остальные 240 - служебные.

Cleanup orphaned /32 (admin)

Если раньше удаляли правила вручную и остались «осиротевшие» /32 (autoupdater не может их убрать, так как родительского домена уже нет), откройте /admin/exit-rules/cleanup. Там:

  • Анализ текущего состояния: разные device_id, разные hostname, ожидающие удаления /32, ожидающие backfill device_ip.
  • Кнопка «Apply» - объединяет дублирующиеся device_id под один канонический (по hostname) и дозаполняет device_ip для всех правил.
  • Идемпотентно: повторный запуск не делает ничего, если данные уже вычищены.

Типичные грабли

  • target_type=ip без /32 - sky-gate автоматически добавляет /32 (например 5.5.5.5 → 5.5.5.5/32), чтобы headscale approve-routes не падал с parsing route: no '/'.
  • Общий IP у двух доменов (например gcr.io и ghcr.io иногда резолвятся в один IP) - sky-gate хранит одну строку /32, без дублей. autoupdater не удалит этот IP, пока он нужен любому из доменов.
  • DNS не резолвится при добавлении домена - правило сохраняется как target_type=domain, autoupdater попробует при следующем цикле (до 5 минут).
  • Cloudflare Challenge на российских сайтах (rutracker и др.) - используйте exit-узел в РФ (sharlotta), а не за рубежом.

Источники IP-диапазонов

Многие сервисы публикуют актуальные IP-диапазоны. Примеры:

  • Сервис A: https://example.com/ips.txt
  • Сервис B: https://example.com/ipranges.json
  • Провайдер C: https://example.com/ips-v4

IP-диапазоны периодически меняются. Рекомендуется:

  • Для подсетей — обновлять раз в 1-3 месяца вручную через AI.
  • Для доменов — autoupdater обновляет IP автоматически, ничего делать не нужно.
